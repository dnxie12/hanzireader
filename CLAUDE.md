# Hanzi Reader

Chinese character reading flashcard PWA for heritage Mandarin speakers. Teaches character recognition (not writing) using spaced repetition. No backend — fully static site deployed to GitHub Pages.

## Architecture

- **Vanilla JS**, no framework, no build step, no bundler
- Hash-based routing with 5 screens: Home (`#home`), Read (`#read`), Study (`#study`), Browse (`#browse`), Stats (`#stats`)
- Data: 2,400 characters merged from hanziDB + Make Me a Hanzi + CC-CEDICT
- SRS: FSRS v6 via ts-fsrs v5.2.3 (vendored UMD bundle at `js/vendor/fsrs.umd.js`, with hardcoded fallback scheduling)
- Storage: localStorage with in-memory cache (keys: `hanzi_srs`, `hanzi_progress`)
- PWA: service worker (cache-first), manifest.json
- Offline-first: all data pre-bundled, no runtime API calls
- **First-visit flow**: first-time visitors (no localStorage) see the placement test overlay (`js/placement.js`) before any other screen. Crawlers and verification bots will see this, not the home screen.

## Script Loading Order (Critical)

Scripts must load in this order (see index.html):

1. `data/char_data.js` — defines `window.CHAR_DATA`, `window.LEARN_ORDER`, `window.RADICAL_DATA`
2. `data/snippets.js` — defines `window.SNIPPET_DATA`
3. `js/storage.js` — no dependencies
4. `js/data.js` — depends on CHAR_DATA, Storage
5. `js/ui.js` — depends on Data, Storage
6. `js/vendor/fsrs.umd.js` — defines `window.FSRS` (ts-fsrs library)
7. `js/srs.js` — depends on Storage, optional window.FSRS
8. `js/home.js`, `js/read.js`, `js/study.js`, `js/browse.js`, `js/stats.js` — screen modules
9. `js/app.js` — router, depends on all screens

## Data Schemas

### Character Data (`window.CHAR_DATA`)

Object keyed by character. Each entry:

```js
{ f: 1,           // frequency rank (1-2400)
  p: "mā",        // pinyin with tone marks
  d: "mother",    // English definition
  r: "女",         // radical
  s: 6,           // stroke count
  h: 1,           // HSK level
  dc: "⿰女马",    // decomposition
  et: "ps",       // etymology type: p=pictographic, ic=ideographic, ps=phono-semantic
  eh: "...",       // etymology hint
  cw: [["妈妈","māma","mother"], ...] } // compound words [chars, pinyin, def]
```

### SRS State (`localStorage: hanzi_srs`)

Object keyed by character. Each entry:

```js
{ due: "ISO date",  stability: N, difficulty: N,
  elapsed_days: N,  scheduled_days: N, reps: N, lapses: N,
  state: 0-3,       // 0=new, 1=learning, 2=review, 3=relearning
  last_review: "ISO date",
  learning_steps: N }
```

### Progress (`localStorage: hanzi_progress`)

```js
{ streak: { current: 0, longest: 0, lastDate: null },
  daily: { "2026-03-14": { reviews: 10, newCards: 2, correct: 8, timeMs: 0, snippetsRead: 0 } },
  settings: { newPerDay: 25, theme: "system", placementDone: false, currentIndex: 0 },
  readingFlags: [],           // chars flagged for priority review from reading
  readHistory: {} }           // { snippetId: "YYYY-MM-DD" }
```

## Reading Practice (Snippets)

Contextual reading exercises that complement SRS flashcard study. Users read short 3-sentence Chinese passages selected to match their proficiency, reinforcing characters they've learned in natural context.

### User flow

1. **Unlock**: Requires 20+ known characters. Passages individually unlock at 60% character coverage.
2. **Snippet list**: Passages grouped by difficulty (Beginner / Intermediate / Advanced / Expert), each showing a text preview, coverage %, topic, and read status (checkmark).
3. **Start session**: Selects 5 optimal passages via scoring algorithm, or tap a single passage to read it standalone.
4. **Reading view**: Large Chinese text with visual indicators — known characters (solid underline), learning characters (learning-color underline), unseen characters (dotted underline), punctuation (dimmed). Tap any character to see pinyin (tone-colored), definition, and buttons to flag it for priority SRS review ("+ Review") or open the full detail modal ("More").
5. **Session summary**: Shows passages read, characters encountered, lookups made, duration, and count of flagged characters. Navigates to home on dismiss.

### Snippet selection algorithm

Five weighted factors score each passage, with hard gates rejecting unsuitable ones:

| Factor | Weight | Goal |
|---|---|---|
| Coverage | 30% | Peak score at 90% known chars (i+1 sweet spot: 85–95%) |
| Reinforcement | 35% | Prefer passages containing due/near-due SRS characters |
| Stretch | 20% | Best with 1–2 unknown chars; penalizes 0 or 4+ |
| Diversity | 10% | Reward variety of radicals among known chars |
| Freshness | 5% | Prefer passages not read in past 7 days |

**Hard gates** (instant reject): <75% coverage, >2.5 unknown chars per sentence, read within past 7 days.

Topic diversity: sessions include at most 4 passages from the same topic before filling greedily.

### Flagging integration

Characters flagged during reading are stored in `readingFlags` (via `Storage.addReadingFlag()`). They feed back into the SRS study queue for priority review. Flags are consumed by `Storage.consumeReadingFlags()`.

### Snippet data generation

Snippets are generated offline via `build/build_snippets.py` using the Claude API. Passages are organized into vocabulary buckets (cumulative character thresholds), 20 snippets per bucket, across 8 topic categories (everyday, family, food, weather, travel, nature, work, society). Output is cached in `data/snippets.js` as `window.SNIPPET_DATA`. Each snippet has: `id` (MD5 hash), `text`, `chars` (unique characters), `sentenceCount`, `staticDiff` (1–4), `topic`.

To regenerate: `ANTHROPIC_API_KEY=sk-... python3 build/build_snippets.py`

## Browse Screen

Searchable grid of all 2,400 characters with lazy-loaded pagination.

### Features

- **Search**: by character, pinyin (accent-agnostic), or English meaning. Debounced at 150ms.
- **Search mode toggle**: cycles All / Pinyin / Meaning to narrow matching.
- **Filters**: SRS state (All / New / Learning / Known / Relearning) + top 25 radicals (those with 5+ characters).
- **Detail modal**: tap any character for full info (pinyin, definition, radical, strokes, frequency, etymology, decomposition, compounds).

### Lazy loading

The grid renders in batches of 200 characters (`BATCH_SIZE = 200`) to avoid rendering all 2,400 DOM elements at once. A "Load More (X remaining)" button appears when there are more characters to show. Each click appends the next 200. The button disappears once all characters are visible. On tab switch back, only SRS state CSS classes are refreshed (via `refreshGridStates()`) — no full re-render.

### State color coding

Grid items get a CSS class based on SRS state: `new` (purple), `learning` (amber), `review` (green), `relearning` (red). Updated on every render and tab switch.

## Study Flow

1. **New Card Phase**: Show character + pinyin + definition + compounds → Quiz (pick correct char from 4 options) → If correct: rated "Good", added to mini-review queue
2. **Review Phase**: Progressive reveal (tap to show: pinyin → definition → compounds) → Rate: Again/Hard/Good/Easy
3. **Mini-reviews**: Newly learned cards re-appear after a few cards for reinforcement
4. **Session end**: Summary shown, streak updated

## SRS Scheduling

### ts-fsrs integration

The app uses ts-fsrs v5.2.3, vendored locally at `js/vendor/fsrs.umd.js` (UMD bundle, ~73KB). The library exposes `window.FSRS` globally. Key API usage in `js/srs.js`:

- `new window.FSRS.FSRS({})` — creates the scheduler instance (default parameters, 90% target retention)
- `window.FSRS.createEmptyCard()` — creates a new card object (returns plain object with `due`, `stability`, `difficulty`, `elapsed_days`, `scheduled_days`, `reps`, `lapses`, `learning_steps`, `state`, `last_review`)
- `fsrs.repeat(card, now)` — computes next state for all 4 ratings. Returns `{ 1: { card }, 2: { card }, 3: { card }, 4: { card } }` keyed by Rating enum (1=Again, 2=Hard, 3=Good, 4=Easy)

### Fallback scheduler

If `window.FSRS` is undefined (vendor file fails to load), `srs.js` falls back to a simple interval scheduler with fixed multipliers. Also used as a per-card safety net via try-catch around `fsrs.repeat()`. Base intervals: Again=1min, Hard=10min, Good=1day, Easy=4days, scaled by a stability multiplier that grows ×1.8 (Good) or ×2.5 (Easy) per review.

### Card state machine

```
new(0) → learning(1) → review/known(2)
                ↑              ↓
          relearning(3) ←──────┘  (on "Again" rating)
```

## Key Conventions

- All data rendered to innerHTML MUST be escaped via `UI.esc()` to prevent XSS
- CSP header restricts scripts to self-only
- Service worker cache version in `sw.js` must be bumped when deploying changes
- Default 25 new cards/day, slider range 5-50
- Tone colors: tone 1 (red), tone 2 (orange), tone 3 (green), tone 4 (blue), tone 5/neutral (gray)
- SRS state colors: new (purple), learning (amber), known (green), relearning (red)
- Literacy % is frequency-weighted (knowing common chars counts more)
